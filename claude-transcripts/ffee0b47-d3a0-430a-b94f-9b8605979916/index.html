Systematic Debugging

Overview

Random fixes waste time and create new bugs. Quick patches mask underlying issues.

Core principle: ALWAYS find root cause before attempting fixes. Symptom fixes are failure.

Violating the letter of this process is violating the spirit of debugging.

The Iron Law

NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST

If you haven't completed Phase 1, you cannot propose fixes.

When to Use

Use for ANY technical issue: - Test failures - Bugs in production - Unexpected behavior - Performance problems - Build failures - Integration issues

Use this ESPECIALLY when: - Under time pressure (emergencies make guessing tempting) - "Just one quick fix" seems obvious - You've already tried multiple fixes - Previous fix didn't work - You don't fully understand the issue

Don't skip when: - Issue seems simple (simple bugs have root causes too) - You're in a hurry (rushing guarantees rework) - Manager wants it fixed NOW (systematic is faster than thrashing)

The Four Phases

You MUST complete each phase before proceeding to the next.

Phase 1: Root Cause Investigation

BEFORE attempting ANY fix:

  1. Read Error Messages Carefully
  2. Don't skip past errors or warnings
  3. They often contain the exact solution
  4. Read stack traces completely

  5. Note line numbers, file paths, error codes

  6. Reproduce Consistently

  7. Can you trigger it reliably?
  8. What are the exact steps?
  9. Does it happen every time?

  10. If not reproducible → gather more data, don't guess

  11. Check Recent Changes

  12. What changed that could cause this?
  13. Git diff, recent commits
  14. New dependencies, config changes

  15. Environmental differences

  16. Gather Evidence in Multi-Component Systems

WHEN system has multiple components (CI → build → signing, API → service → database):

BEFORE proposing fixes, add diagnostic instrumentation: ``` For EACH component boundary: - Log what data enters component - Log what data exits component - Verify environment/config propagation - Check state at each layer

Run once to gather evidence showing WHERE it breaks THEN analyze evidence to identify failing component THEN investigate that specific component ```

Example (multi-layer system): ```bash # Layer 1: Workflow echo "=== Secrets available in workflow: ===" echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

# Layer 2: Build script echo "=== Env vars in build script: ===" env | grep IDENTITY || echo "IDENTITY not in environment"

# Layer 4: Actual signing codesign --sign "$IDENTITY" --verbose=4 "$APP" ```

All four investigation threads are complete and verified against two ElevenLabs API reference pages plus a local execution of the repo's actual mapping code. Here's the full report.

TL;DR — the styles never reach the model

Your instinct is right, and the docs make it concrete: we pin model_id: "music_v2", but we put every style token in the music_v1 schema's location. Per the current docs (both /v1/music and /v1/music/plan reference pages agree), the v2 CompositionPlan is { chunks: [...] } only — positive_global_styles / negative_global_styles exist solely on the v1 MusicPrompt shape. Meanwhile the per-chunk positive_styles / negative_styles — which the docs call out as "the most important as they set the overall tone and genre" (they recommend 6–7 on the first chunk) — are the fields we send empty, deliberately (mapping.ts:27). context_adherence is likewise a per-chunk field (documented default "high"), not the top-level key we send it as.

So the bytes we POST contain all the DNA direction, but in keys the documented v2 schema doesn't define, while the keys v2 actually reads carry nothing but lyrics. The API tolerates the extra keys and returns 200 (your stored fixtures have real x-trace-id headers, so this passed in production) — the classic silent failure you predicted. The only DNA field that demonstrably reaches the audio is lyricalObsessions, laundered through Claude's lyric seed into chunk text. Cordelia sounds gospel-ish because her lyrics say "pew," "prayer," and "Lord" — not because any slider worked.

One caveat on confidence: the docs define the contract; I can't prove from docs alone that the server drops rather than leniently merges the v1 keys. The empirical confirmation is exactly the SPEC's "hour one" test (one artist, one slider at each extreme, listen + spectral rolloff), and scripts/ab-test.ts exists for it.

4. Style tokens: positive vs negative across all 36 stored fixture plans

Average 21.7 positive vs 14.0 negative per plan (neg/pos = 0.65). Docs recommend ~6–7 positives and say empty negatives are typical — so if these arrays were ever moved to the chunk where they belong as-is, we'd be 3× over the style budget and heavily over-negated. Specific problems in the negative design:

  • Symmetric negation amplifies mild opinions into hard bans. mapBipolarAxis emits as many negative tokens as positive for the same magnitude — Cordelia's barely-off-center improvisedStructured: -0.2 bans "structured songwriting" outright; her loudQuiet: -0.5 bans "quiet" and "hushed dynamics" on a gospel-soul artist who presumably has dynamics.
  • mapVocalAxis negatives ignore magnitude entirely (opposing.slice(1) at any strength past the deadzone), so a 0.16 lean produces two hard vocal bans.
  • Vocal band boundary bug: whispersScreams > 0.55 takes two tokens, so Cordelia at 0.6 gets "screamed vocals" as a positive style. A soul belter at 0.56 and a metal screamer at 0.95 get nearly the same positive list.
  • Fixtures are stale relative to the current vocabulary: modem-candy's stored negatives still contain "live drums" (removed from styleTokens.ts per its comment), and stored Cordelia plans (22/15) differ from a fresh build (23/16).

5. Lyric length vs duration

Across all 36 plans: 120–157 chars, 20–31 words per 30 000 ms chunk → 0.67–1.03 words/sec (avg ≈ 0.85). Typical sung pop delivery is ~1.5–3 words/sec, so we're supplying roughly 10–15 seconds of singable material for a 30-second track — the model has to pad with instrumental sections or drag delivery. The payloads are "correct" per the self-imposed 170/180-char cap, but the cap itself looks like a misreading: the empirically discovered "201 chars → 500" (SPEC) matches the documented 200-chars-per-line limit (v1 allows 30 lines × 200 chars per section). Newline-separated multi-line text — optionally with [Verse]/[Chorus] labels, which the v2 text field explicitly supports — should lift the ceiling ~30×.

6. Quantization audit

  • Sonic sliders (CreateStudio.tsx:331): step 0.05, fine — but NEUTRAL_DEADZONE = 0.15 silently zeroes anything at ±0.1 (one slider notch off-center = two notches, 0.10, still dead). Hit in 7 fixture axes, most often darkHopeful.
  • Influence weights: yes, weights round to zero. UI step is 0.01 with renormalization (continuous), but influenceTokens does Math.round(weight * 4) → any weight < 0.125 emits nothing. Four of twelve fixtures have a genre contributing zero tokens (danny-halloran "blue-collar anthemic pop" 0.10, modem-candy "vaporwave" 0.10, solange-vaugirard "nu-disco" 0.10, verse-halloway "dubstep" 0.12). Only the single heaviest genre is protected by the lead-genre backstop. The UI still shows these as e.g. "10%", so the control looks live but isn't.
  • Vocal pad: continuous pointer input, then banded at 0.55 / 0.8 — the 0.55 boundary is the "screamed vocals" cliff above; the 0.15 deadzone also killed silas-crowder's 0.1 whisper lean.

Why Order Matters

"I'll write tests after to verify it works"

Tests written after code pass immediately. Passing immediately proves nothing: - Might test wrong thing - Might test implementation, not behavior - Might miss edge cases you forgot - You never saw it catch the bug

Test-first forces you to see the test fail, proving it actually tests something.

"I already manually tested all the edge cases"

Manual testing is ad-hoc. You think you tested everything but: - No record of what you tested - Can't re-run when code changes - Easy to forget cases under pressure - "It worked when I tried it" ≠ comprehensive

Automated tests are systematic. They run the same way every time.

"Deleting X hours of work is wasteful"

Sunk cost fallacy. The time is already gone. Your choice now: - Delete and rewrite with TDD (X more hours, high confidence) - Keep it and add tests after (30 min, low confidence, likely bugs)

The "waste" is keeping code you can't trust. Working code without real tests is technical debt.

"TDD is dogmatic, being pragmatic means adapting"

TDD IS pragmatic: - Finds bugs before commit (faster than debugging after) - Prevents regressions (tests catch breaks immediately) - Documents behavior (tests show how to use code) - Enables refactoring (change freely, tests catch breaks)

"Pragmatic" shortcuts = debugging in production = slower.

"Tests after achieve the same goals - it's spirit not ritual"

No. Tests-after answer "What does this do?" Tests-first answer "What should this do?"

Tests-after are biased by your implementation. You test what you built, not what's required. You verify remembered edge cases, not discovered ones.

Tests-first force edge case discovery before implementing. Tests-after verify you remembered everything (you didn't).

30 minutes of tests after ≠ TDD. You get coverage, lose proof tests work.
